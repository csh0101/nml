# 卷03: 场景执行与验收闭环

## 1. 为什么必须是“闭环”
很多人做 DSL 只做到“能生成”，但没有“验收和重写”。结果是:
- 第一次看着还行
- 连续跑 20 章后风格漂移
- 问题难定位，只能人工返工

NML 要解决的是生产问题，不是一次性创作演示。所以必须是闭环:
输入 -> 执行 -> 验收 -> 重写 -> 通过。

## 2. 闭环中的角色分工
### 2.1 输入层（你写）
由标签定义任务边界:
- `章节蓝图`: 章节目标和钩子
- `场景`: 当前执行单元
- `任务`: 本场景核心目标
- `信息揭示`: 必须/禁止信息
- `风格模板`: 文风硬约束

### 2.2 执行层（Agent）
Agent 只做一件事: 在约束内生成当前场景文本。

### 2.3 验收层（规则）
验收器只判断“是否合规”，不做审美争论。

### 2.4 修正层（Agent + 规则）
失败时返回失败原因，按原因重写，不是整章推翻。

## 3. 为什么“场景”是最小执行单元
如果一次让 Agent 写整章:
- 上下文太长，约束容易失效
- 局部失败会拖累全章重写

按场景执行的优势:
- 失败定位精准
- 重试成本低
- 可并行（多个场景分工）

## 4. 场景执行标准流程
### 4.1 读取
读取当前场景 + 必要上文，不加载无关章节。

### 4.2 校验引用
先查 `地点引用` 和 `视角角色` 是否存在。引用失败直接停止，不生成。

### 4.3 抽取约束
按优先级合并:
1. `写作准则`
2. `风格模板`
3. `场景`属性
4. `任务`与`段落计划`

### 4.4 生成
按 `段落计划` 逐段生成，确保每段完成明确功能。

### 4.5 验收
按 `验收规则/硬性失败` 判定是否通过。

### 4.6 重写
仅针对失败项重写相关段，不改已通过段落。

## 5. 验收要“可计算”，不要“靠感觉”
错误示例:
- “感觉不够紧张”

正确示例:
- 单句字数超过上限
- 出现禁用词
- 未覆盖必须提及项
- 结尾未命中场景收束落点

前者不可自动化，后者可自动化。

## 6. 你最该先落地的 6 条硬性失败
1. 出现任一禁用词。
2. 单句长度超过上限。
3. 任一段未新增信息点。
4. 必须提及项缺失。
5. 禁止提及项出现。
6. 场景结尾未命中 `场景收束.落点`。

这 6 条足够把“跑偏”压到很低。

## 7. 常见闭环失效点
### 7.1 输入模糊
`任务` 写得太空，导致生成散。

### 7.2 规则冲突
比如 `字数目标很短` + `必须提及太多`，会迫使文本失真。

### 7.3 失败反馈不可执行
只说“重写一下”没有意义，必须给出具体失败项。

### 7.4 重写粒度太大
整章重写会把已合格段落一起破坏。

## 8. 面向实战的建议
- 先固定一个风格模板，连续跑 5 章，别频繁换模板。
- 每章先只开 3 个场景，避免结构过重。
- 先追求“稳定合格”，再追求“文学惊艳”。

## 9. 本卷小结
你的 NML 项目真正的护城河，不是标签数量，而是闭环能力:
- 生成有边界
- 验收有标准
- 失败可定位
- 重写可控

这四点建立后，你的系统才会越跑越稳，而不是越跑越散。
