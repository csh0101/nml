# 卷05: 从标签定义到解析器实现的最小技术路线

## 1. 目标边界
本卷只回答一个问题:
如何把你已经定义好的 NML 标签，落地为一个可运行 MVP。

不做大而全，不做引擎级产品，只做:
- 能解析
- 能校验
- 能输出给 Agent 执行
- 能根据验收结果回写失败原因

## 2. MVP 组件清单
最小实现建议 4 个组件:
1. `Loader`：加载 NML 文件文本。
2. `Validator`：结构与语义校验。
3. `Planner`：把场景转换为执行输入。
4. `Runner`：调用 Agent，执行生成和重试。

你可以把这四个组件理解为“最小写作流水线”。

## 3. 文件与目录建议
建议在项目里先固定这套目录:
- `spec/` 放标签定义（你已经有 docs，可镜像）
- `examples/` 放可运行样例 NML
- `runtime/` 放解析与执行代码
- `reports/` 放每次验收报告

关键是“样例和报告”必须落盘，方便回归对比。

## 4. 解析顺序（必须固定）
### 第一步：结构解析
先把 XML/NML 读成树结构。

### 第二步：基础校验
校验必填标签与必填属性。

### 第三步：引用校验
校验 `地点引用 -> 地点.id`、`视角角色 -> 角色.id`。

### 第四步：执行计划生成
按 `章节 -> 场景 -> 段落计划` 生成可执行队列。

### 第五步：逐场景执行
一次只执行一个场景，失败可重试。

不要把顺序打乱，否则问题定位会非常痛苦。

## 5. 校验分层策略
### 5.1 结构层
检查标签存在性与层级合法性。

### 5.2 引用层
检查 id/ref 是否命中。

### 5.3 规则层
检查配比、字数容差、禁用词、必须提及等。

这三层要分开报错，才能知道是“文件坏了”还是“内容违规”。

## 6. 错误码设计（先简后全）
建议先定一组稳定错误码:
- `E1xxx` 结构错误
- `E2xxx` 引用错误
- `E3xxx` 规则错误
- `E4xxx` 执行错误

示例:
- `E2001 InvalidLocationRef`
- `E3004 MissingRequiredInfo`
- `E3007 ForbiddenInfoFound`

错误码一旦对外使用，尽量不要频繁改名。

## 7. 执行输入契约（给 Agent）
不要把整份项目原文直接塞给 Agent。

每次执行只给:
- 全局约束（写作准则 + 风格模板）
- 当前章节蓝图
- 当前场景
- 当前段落计划
- 当前验收规则

输入越聚焦，输出越稳定。

## 8. 重试机制怎么做
重试不要“整场景重跑无脑覆盖”。

建议:
1. 先产出初稿。
2. 验收得到失败项列表。
3. 只把失败项和相关段落喂回 Agent。
4. 合并修订段，重新验收。

重试上限建议先设 2-3 次。

## 9. 报告机制（非常关键）
每次执行都写一份报告到 `reports/`，至少包含:
- 场景 id
- 输入摘要（关键约束）
- 失败项/通过项
- 最终结果版本号

没有报告，你无法做稳定迭代。

## 10. 新手最容易踩的实现坑
1. 先写执行器，后补校验器。
2. 一次跑整章，不做场景切分。
3. 错误只打日志，不结构化。
4. 没有固定样例，改一次坏一次。

正确顺序是: 样例 -> 校验 -> 场景执行 -> 重试 -> 报告。

## 11. 一周落地计划（MVP）
- Day1: 固定标签定义与 2 个样例。
- Day2: 完成结构与引用校验。
- Day3: 完成规则校验（最小 6 条硬性失败）。
- Day4: 完成单场景执行输入拼装。
- Day5: 接入 Agent 调用与重试。
- Day6: 加执行报告。
- Day7: 跑 3 章回归测试。

## 12. 本卷小结
你要的是可运营的写作系统，不是一次性 demo。

只要先把“解析-校验-执行-重试-报告”这条最小链路打通，后面再扩展标签和高级功能都不会乱。
