# 卷06: 标签优先级与冲突处理

## 1. 为什么必须先定义优先级
没有优先级时，系统会出现同一输入两种结果:
- A 轮按 `任务` 写
- B 轮按 `风格模板` 写

这会让重试和回归测试失效。

所以你必须在规范里明确: 冲突时听谁。

## 2. 建议的总优先级（高 -> 低）
1. 安全与保密约束（禁止提及、硬性失败）
2. 全局硬约束（写作准则）
3. 风格模板约束（句法/词汇/推进/对话/收束）
4. 场景级约束（节奏、氛围、字数目标、视角角色）
5. 任务目标
6. 段落计划细项
7. 自由描述文本

这套顺序能保证“先不违规，再求好看”。

## 3. 冲突类型与裁决
### 3.1 风格 vs 任务
例子:
- 任务希望“长段抒情”
- 风格模板限制“单句上限 24、抒情段上限 10%”

裁决: 保留任务核心信息，改写表达方式，服从风格约束。

### 3.2 字数 vs 信息量
例子:
- 场景字数目标 300
- 必须提及 8 条

裁决: 先满足必须提及，再触发“字数超容差警告”；必要时拆分场景。

### 3.3 配比 vs 段落计划
例子:
- 配比要求动作 40%
- 段落计划几乎全是环境段

裁决: 优先满足配比下限，段落计划自动重排。

### 3.4 必须提及 vs 禁止提及
例子:
- 同一信息同时被列入必须与禁止

裁决: 直接判定配置冲突，返回错误并停止执行。

## 4. 冲突处理状态
建议定义三种处理状态:
- `BLOCK`：不可调和冲突，停止执行。
- `DEGRADE`：可降级执行，记录警告。
- `AUTO_FIX`：系统可自动改写并继续。

示例:
- 引用不存在 => `BLOCK`
- 字数轻微超容差 => `DEGRADE`
- 任务措辞与风格冲突 => `AUTO_FIX`

## 5. 最小冲突错误码
- `E3101 HardConstraintConflict`
- `E3102 RequiredVsForbiddenConflict`
- `E3103 BudgetOverflow`
- `E3104 StyleTaskConflict`
- `E3105 InvalidPriorityDefinition`

这些错误码要在验收报告中固定输出，便于回归追踪。

## 6. 可落地的冲突裁决算法（简版）
1. 汇总所有约束并打上优先级。
2. 从高到低逐层合并。
3. 发现冲突时先查是否有自动修正策略。
4. 无策略且影响硬约束 -> `BLOCK`。
5. 有策略且不破坏硬约束 -> `AUTO_FIX`/`DEGRADE`。
6. 输出最终执行约束快照供后续追溯。

## 7. 你现在就能用的规则
先把以下 4 条写死:
1. `禁止提及` 永远高于 `任务描述`。
2. `写作准则.视角` 永远高于局部段落措辞。
3. `风格模板.句法上限` 永远高于“想写得更华丽”。
4. 任何引用缺失直接停止执行。

这 4 条能立刻避免 80% 的跑偏。

## 8. 报告里必须记录什么
每次冲突处理都要记录:
- 冲突双方
- 触发规则
- 最终裁决
- 是否自动修正

没有这 4 项，你以后很难解释“为什么系统这么写”。

## 9. 本卷小结
冲突处理不是异常分支，而是生产主路径。

只要优先级和裁决策略先定义清楚，NML 才能稳定进入工程化阶段。
