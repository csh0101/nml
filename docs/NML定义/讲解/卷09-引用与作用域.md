# 卷09: 引用与作用域

## 1. 为什么这一卷很关键
你这套 NML 一旦开始拆分文件、多人协作，最容易出问题的不是文风，而是引用。

典型问题:
- `地点引用` 指向了不存在的地点
- 两个文件里出现相同 `id`
- 合并导入后引用歧义

所以你必须明确“作用域规则”。

## 2. 三种作用域
### 2.1 全局作用域
在整个 `小说项目` 内唯一的标识。
建议用于:
- `角色.id`
- `地点.id`
- `风格模板.id`

### 2.2 局部作用域
只要求在父节点内唯一。
建议用于:
- `场景.id`（章节内唯一）
- `段.序号`（段落计划内唯一）

### 2.3 临时作用域
仅在一次执行任务中有效，不写回源文件。
建议用于:
- 执行器内部变量
- 本轮重试上下文标识

## 3. 引用关系基线
先固定一张最小引用图:
- `章节.风格模板引用 -> 风格模板.id`
- `场景.地点引用 -> 地点.id`
- `场景.视角角色 -> 角色.id`（建议强校验）

如果这三条都稳定，系统就能正常跑。

## 4. 唯一性规则建议
### 4.1 强唯一（全局）
- `角色.id`
- `地点.id`
- `风格模板.id`

### 4.2 弱唯一（局部）
- `场景.id` 在同章节内唯一
- `段.序号` 在同段落计划内唯一

### 4.3 允许重复
- `章节.标题` 可以重复（不建议）
- `任务.目标` 文本可以重复

## 5. 命名规范（防冲突）
建议统一采用:
- 小写 ASCII
- 下划线分词
- 前缀表达类型

示例:
- `char_lin_xun`
- `loc_broken_temple`
- `style_cold_blade_v1`

你也可以不加前缀，但多人协作时前缀非常省事。

## 6. 跨文件导入后的冲突处理
即使你现在没启用 `导入`，也建议先定义策略。

导入展开后应执行:
1. 扫描全局 id 索引。
2. 发现重复 id -> 直接报错 `BLOCK`。
3. 禁止“自动重命名”源文件 id（会破坏可追溯性）。

自动重命名只可用于运行时副本，不可回写源文件。

## 7. 引用校验执行顺序
建议固定为:
1. 建立符号表（角色/地点/模板）。
2. 校验所有引用字段。
3. 输出引用错误列表。
4. 若有错误，停止执行生成。

不要在引用未通过时继续调用 Agent。

## 8. 错误码建议
- `E2101 DuplicateGlobalId`
- `E2102 DuplicateLocalId`
- `E2201 MissingRoleRef`
- `E2202 MissingLocationRef`
- `E2203 MissingStyleTemplateRef`
- `E2204 AmbiguousReference`

## 9. 多人协作下的实践规则
1. 新增实体前先查 id 索引。
2. 任何重命名必须全局替换并过校验。
3. 每次合并前执行一次“仅引用校验”。

引用问题越早挡住，后面返工越少。

## 10. 本卷小结
引用与作用域是 NML 工程化的底层稳定器。

你可以容忍文风争议，但不能容忍引用漂移。前者影响质量，后者影响系统可运行性。
