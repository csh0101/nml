# 卷12: 回归测试样例集

## 1. 为什么要做回归样例集
没有固定样例集，你每次改规则都不知道:
- 修了旧问题没有
- 引入新问题没有

回归样例集就是“长期体检表”，不是一次性测试。

## 2. 样例集的最小组成
建议最少 3 组:
1. `happy_path`：标准合规样例，应该稳定通过。
2. `failure_path`：故意违规样例，应该稳定失败。
3. `edge_path`：边界样例，验证容错与临界值。

## 3. 每组样例该覆盖什么
### 3.1 happy_path
覆盖主流程:
- 引用完整
- 任务明确
- 必须提及覆盖
- 结尾命中场景收束

目标: 保证“正常项目不会被误杀”。

### 3.2 failure_path
每条硬规则至少一个失败样例:
- 缺失 `地点引用`
- 出现 `禁止提及`
- 单句超上限
- 缺失 `必须提及`

目标: 保证“违规一定被拦截”。

### 3.3 edge_path
覆盖临界值:
- 字数刚好命中上下容差
- 长句占比刚好等于上限
- 最小可行标签集

目标: 保证“边界行为可预测”。

## 4. 样例文件命名建议
建议统一命名:
- `hp_001_basic_scene.nml`
- `fp_003_missing_location_ref.nml`
- `ep_002_ratio_boundary.nml`

文件名自带意图，方便快速定位失败原因。

## 5. 预期结果文件（必须）
每个样例配一个预期结果文件:
- 通过样例: 预期 `PASS`
- 失败样例: 预期错误码列表

示例:
`fp_003_missing_location_ref.expect.json`

没有预期文件，就无法自动判定回归。

## 6. 回归执行流程
1. 加载样例。
2. 执行解析/校验/生成/验收。
3. 与 `.expect.json` 对比。
4. 产出回归报告。

结果分档:
- `REG_PASS`
- `REG_FAIL_EXPECT_MISMATCH`
- `REG_FAIL_RUNTIME_ERROR`

## 7. 你最该优先建的 8 个样例
1. 标准单场景通过样例。
2. 缺失地点引用失败样例。
3. 禁止提及命中失败样例。
4. 必须提及缺失失败样例。
5. 长句超上限失败样例。
6. 结尾未命中收束失败样例。
7. 字数容差边界样例。
8. 仅最小标签集合样例。

这 8 个就足够让 MVP 稳定运行。

## 8. 回归阈值建议
除了硬规则，还建议加“退化阈值”:
- 长句占比波动 > 5% -> warning
- 必须提及覆盖率下降 -> fail
- 平均句长漂移 > 3 字 -> warning

让你尽早发现风格滑坡。

## 9. CI 接入建议（后续）
当你准备上 CI 时:
- 每次提交必须跑样例集
- 新增规则必须新增失败样例
- 修 bug 必须补一个防回归样例

这三条是长期稳定的关键。

## 10. 本卷小结
回归样例集是“系统不会越改越坏”的保险丝。

你现在做的是写作工程，不是灵感实验。工程一定要有回归基线。
