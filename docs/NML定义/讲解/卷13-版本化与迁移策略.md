# 卷13: 版本化与迁移策略

## 1. 为什么要先定版本策略
没有版本策略，规范一改就会出现:
- 老项目突然跑不通
- 新旧规则混杂
- 团队无法确认当前口径

所以版本化不是发布动作，而是稳定性机制。

## 2. 版本号建议
采用语义化版本思路:
- `主版本.次版本`（如 `2.1`）
- 可附阶段后缀（如 `2.1-M1`）

定义:
- 主版本变更: 允许破坏兼容
- 次版本变更: 向后兼容增量

## 3. 兼容性规则（建议写死）
### 3.1 次版本（2.1 -> 2.2）
允许:
- 新增可选标签
- 新增可选属性
- 新增 enum 值（需默认行为）

禁止:
- 删除已有标签
- 改变已有属性含义
- 把可选改成必填

### 3.2 主版本（2.x -> 3.0）
允许破坏性变更，但必须:
- 提供迁移文档
- 提供迁移工具或迁移规则
- 给出冻结窗口和停用计划

## 4. 文档中的版本声明
建议根标签强制带版本:
- `小说项目 版本="2.1-M1"`

运行时行为:
- 解析器按版本加载对应规则集
- 未声明版本直接失败或按默认版本警告

## 5. 迁移策略三件套
每次升级都要提供:
1. `CHANGELOG`：变更点和影响面。
2. `MIGRATION GUIDE`：如何改旧文件。
3. `COMPAT MATRIX`：各版本支持矩阵。

少一个，升级都会混乱。

## 6. 自动迁移与手动迁移
### 6.1 自动迁移适用
- 字段重命名
- 默认值补齐
- 结构不变的语法修正

### 6.2 手动迁移适用
- 语义变更
- 标签拆分/合并
- 规则冲突需要业务判断

原则: 能自动的自动，涉及语义判断的一律人工确认。

## 7. 迁移流程建议
1. 备份原文件。
2. 运行迁移器生成新版本草稿。
3. 执行结构/引用/规则校验。
4. 跑回归样例。
5. 产出迁移报告并人工抽检。

迁移完成后再切换默认运行版本。

## 8. 版本错误码建议
- `E5101 UnsupportedNmlVersion`
- `E5102 MissingVersionDeclaration`
- `E5103 MigrationRequired`
- `E5104 MigrationFailed`

## 9. 过渡期策略
建议保留“兼容窗口”:
- 新版本发布后，旧版本继续支持一段时间
- 在报告中标记旧版本“即将停用”警告

这样不会把团队一次性推入风险区。

## 10. 本卷小结
版本化的目标不是“看起来专业”，而是“升级不炸”。

你做的是长期系统，必须让旧项目可维护、新项目可迭代、团队可协作。
