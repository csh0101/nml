# 卷15: 从讲解稿到正式标准

## 1. 为什么要做“收敛”
分卷讲解解决了理解问题，但还不是“标准文档”。

正式标准必须具备:
- 单一权威入口
- 可引用条款编号
- 可执行附录（错误码、类型表、契约表）

否则团队会继续“各自理解”。

## 2. 建议的标准文档结构
建议最终发布为一份主标准 + 若干附录。

### 主标准（Normative）
1. 范围与目标
2. 术语定义
3. 标签语法
4. 语义优先级
5. 执行流程
6. 验收规则
7. 版本与兼容策略

### 附录（Informative/Reference）
- A: 错误码字典
- B: 属性类型语法
- C: 最小示例
- D: 迁移指南
- E: 回归样例清单

## 3. 规范条款编号
建议所有强制条款编号化，例如:
- `NML-CORE-001`: 文档必须且仅有一个根标签
- `NML-REF-003`: 场景地点引用必须命中地点 id
- `NML-VAL-007`: 出现禁止提及项时必须判定失败

有编号后，你在报告里可直接引用条款，沟通成本大幅下降。

## 4. 强制语气统一
建议统一使用:
- `MUST`（必须）
- `SHOULD`（建议）
- `MAY`（可选）

不要混用“最好、尽量、建议考虑”等模糊词。

## 5. 从当前文档到正式标准的收敛步骤
1. 提取每卷中的“必须规则”。
2. 去重并合并冲突表述。
3. 补齐编号与术语定义。
4. 生成主标准草案。
5. 用 3 组样例跑一致性审查。
6. 冻结 v2.1 标准文本。

## 6. 一致性审查清单
发布前至少检查:
- 标签定义与错误码是否一一对应
- 规则是否能被程序判定
- 例子是否符合规范本身
- 版本策略是否可执行

如果某条规则无法编程判定，就不算正式标准条款。

## 7. 标准与实现分离
必须保持:
- 标准文档描述“应该怎样”
- 解析器代码实现“具体怎么做”

不要把某次实现细节直接写进核心条款，否则后续重构会受限。

## 8. 发布与变更机制
建议给标准文档建立发布流程:
1. 草案（Draft）
2. 候选（RC）
3. 正式（Release）

每次变更必须附:
- 变更摘要
- 影响范围
- 迁移建议

## 9. 你现在可以立刻做的动作
1. 先冻结“卷08 + 卷06 + 卷07 + 卷13”作为 v2.1 核心。
2. 把其余卷内容放入附录与实践指南。
3. 产出 `NML-Standard-v2.1.md` 单文件。

这样你就从“知识集合”升级为“可执行标准”。

## 10. 本卷小结
讲解稿的终点不是“写很多”，而是“形成唯一标准入口”。

你现在已经有了足够材料，下一步就是收敛、编号、冻结版本，然后进入实现与迭代。
