# 卷17: 实现任务拆解

## 1. 本卷目标
把标准文档落到“可执行开发计划”。

你可以直接按本卷拆分任务给自己或团队，不再停留在文档层。

## 2. 实现总目标（MVP）
在 2-3 周内交付一条可跑通链路:
- 读取 NML
- 完成结构/引用/规则校验
- 生成场景执行输入
- 调用 Agent
- 输出验收报告

## 3. 模块拆解
### M1. Parser（解析）
交付物:
- 读取 XML/NML
- 生成内部 AST/DOM

验收标准:
- 能正确解析最小示例
- 结构错误可定位到标签路径

### M2. Validator（校验）
交付物:
- 结构校验
- 引用校验
- 规则校验

验收标准:
- 可输出标准错误码（E1/E2/E3）
- 失败样例命中预期错误

### M3. Planner（执行输入组装）
交付物:
- 场景级执行上下文对象
- 约束优先级合并

验收标准:
- 同一输入每次生成相同 planning snapshot

### M4. Runner（执行与重试）
交付物:
- 单场景执行
- 重试机制（按失败项修复）

验收标准:
- 至少支持 2 次重试
- 能避免整章重跑

### M5. Reporter（报告）
交付物:
- JSON 报告文件
- 指标与错误列表

验收标准:
- 字段满足卷11规范
- 可用于回归对比

## 4. 任务排期建议
### Week 1
- Parser + 结构校验
- 引用校验
- 最小样例集接入

### Week 2
- 规则校验
- Planner
- Runner 初版

### Week 3
- Reporter
- 回归测试
- 文档对齐与发布

## 5. 每周里程碑
- 里程碑 A: `can-parse`
- 里程碑 B: `can-validate`
- 里程碑 C: `can-run-scene`
- 里程碑 D: `can-regress`

每个里程碑必须配一个演示样例，不做“口头完成”。

## 6. 最小接口建议
### parse
`parse(input_file) -> ast`

### validate
`validate(ast) -> {errors,warnings}`

### plan
`plan(ast, chapter, scene) -> execution_context`

### run
`run(execution_context) -> output_text`

### report
`report(context, result, validation) -> json`

先保持接口稳定，再优化内部实现。

## 7. 风险与应对
### 风险1: 规则过多导致实现缓慢
应对: 先实现最小 6 条硬性失败。

### 风险2: Agent 输出不稳定
应对: 先固定模板与最小上下文输入。

### 风险3: 文档与实现脱节
应对: 每周对照 `NML-Standard-v2.1.md` 做一致性检查。

## 8. Definition of Done
一个模块完成必须满足:
1. 有代码
2. 有样例
3. 有测试
4. 有文档

缺一项都不算 Done。

## 9. 本卷小结
你现在已经从“定义 DSL”进入“建设系统”的阶段。

最关键的是节奏: 小步快跑、每周可演示、每步可回归。
