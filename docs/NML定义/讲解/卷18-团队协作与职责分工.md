# 卷18: 团队协作与职责分工

## 1. 为什么需要角色分工
NML 项目涉及三类工作:
- 叙事与设定
- 规则与验收
- 工程与运行

如果没有分工，常见结果是:
- 作者改规则
- 工程师改剧情
- 验收标准无人维护

最后所有人都很忙，但系统不稳定。

## 2. 建议的最小角色模型
### 2.1 内容负责人（Editor）
职责:
- 维护章节蓝图、场景任务、信息揭示
- 审核输出内容是否符合叙事目标

不负责:
- 修改核心解析逻辑

### 2.2 规范负责人（Spec Owner）
职责:
- 维护标签定义、错误码、版本策略
- 管理扩展标签生命周期

不负责:
- 直接写章节正文

### 2.3 工程负责人（Runtime Owner）
职责:
- 维护 parser/validator/runner/reporter
- 保证回归测试稳定通过

不负责:
- 定义文学风格本体

### 2.4 质量负责人（QA/Reviewer）
职责:
- 维护样例集和回归基线
- 审核报告质量与退化趋势

## 3. 交付边界
每个角色的交付物应明确:
- Editor: `章节/场景 NML`
- Spec Owner: `标准与迁移文档`
- Runtime Owner: `代码与运行报告`
- QA: `回归结果与问题单`

边界清楚后，问题定位会非常快。

## 4. 变更流程建议
1. 提交变更提案（内容/规则/代码）。
2. 指定影响范围（标签、版本、样例）。
3. 执行回归。
4. 评审通过后合并。

任何影响标准条款的改动，都必须由 Spec Owner 审批。

## 5. 会议机制（轻量）
建议每周两次固定节奏:
- 规则同步会（30 分钟）
- 质量复盘会（30 分钟）

会议目标不是讨论创意，而是确认:
- 规则是否冲突
- 回归是否退化
- 下周优先修复项

## 6. 问题单分类建议
统一问题标签:
- `spec`
- `runtime`
- `content`
- `qa`
- `migration`

这样你能快速看出瓶颈在“文档”还是“实现”。

## 7. 冲突处理机制
当内容目标与规则冲突时:
1. 先按现行标准执行。
2. 若需改标准，走变更提案。
3. 不允许口头改规则直接上线。

一句话: 先稳定生产，再优化规范。

## 8. 绩效指标（团队层）
建议追踪 4 个指标:
- 回归通过率
- 规则变更导致的失败率
- 平均修复时长
- 场景一次通过率

这些比“写了多少字”更能反映系统成熟度。

## 9. 本卷小结
NML 的成功不只靠标签设计，还靠协作机制。

你要把“写作”变成“可协作生产”，角色边界和流程纪律是关键。
